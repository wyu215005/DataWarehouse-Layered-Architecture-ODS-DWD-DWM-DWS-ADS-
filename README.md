## 1. 为什么需要数据仓库分层？

想象一家大型超市。每天有数以万计的商品被购买、退货、上架、下架。如果所有的记录都混在一张表里——收银记录、库存变动、会员积分、供应商对账——会发生什么？

**混乱。难以查询。更难以分析。**

数据仓库的分层架构，就是给这片混乱带来秩序的解决方案。

### 分层的核心价值

| 价值维度 | 无分层的痛点 | 分层后的收益 |
|---------|------------|------------|
| **数据质量** | 原始脏数据直接使用，错误扩散 | 清洗在 DWD 完成，错误被隔离 |
| **开发效率** | 每个需求重复写相同的清洗逻辑 | 公共逻辑沉淀为模型，复用率高 |
| **性能** | 每次查询扫全表，慢且耗资源 | 汇总宽表提前计算，查询秒级响应 |
| **可维护性** | 改一个字段需修改数十个下游 | 分层隔离变化，影响范围可控 |
| **血缘追踪** | 数据从哪来？不知道 | 清晰的分层 = 清晰的数据血缘 |

### 一个直觉类比

```
原材料仓库 (ODS)
    ↓ 质检 + 分类
半成品车间 (DWD)
    ↓ 粗加工
组装工段 (DWM)
    ↓ 精加工 + 汇总
成品仓库 (DWS)
    ↓ 包装出货
门店货架 (ADS)
```

就像制造业的流水线，每个环节都有明确职责，上游出问题不会立刻污染下游。

## 3. ODS — 操作数据存储层 (Operational Data Store)

### 📌 定义

ODS（Operational Data Store，操作数据存储层）是数据仓库的**第一道门**，也称为**贴源层**。它的核心职责是将各业务系统的数据**原汁原味地**搬运并保存到数据仓库中。

### 🎯 核心作用

- **数据缓冲区**：将在线业务库（OLTP）的数据导入离线分析环境，减少对业务库的查询压力
- **历史快照**：通过每日全量或增量同步，保存数据的历史变化轨迹
- **数据审计**：原始数据不修改，为后续数据问题溯源提供依据
- **技术解耦**：上游业务系统的变更只影响到 ODS，不会直接冲击下游分析模型

### 🔑 核心特征

```
┌─────────────────────────────────────────────────────┐
│                  ODS 层核心特征                       │
├─────────────────┬───────────────────────────────────┤
│  贴源性          │ 表结构与源系统保持一致或极小改动       │
│  原始性          │ 不做任何业务逻辑加工                  │
│  完整性          │ 保留全量历史数据（或增量+快照）         │
│  时效性          │ 通常 T+1（次日凌晨跑批）或准实时        │
│  可追溯性        │ 任何数据问题都能在此找到根源            │
└─────────────────┴───────────────────────────────────┘
```

### 📦 典型表命名规范

```
ods_{业务域}_{源表名}_{同步类型}

示例:
ods_trade_order_full        -- 订单表全量快照
ods_trade_order_inc         -- 订单表增量数据
ods_user_userinfo_full      -- 用户信息全量
ods_log_page_view_inc       -- 页面浏览日志增量
```

### 💡 常见数据来源

```
          业务数据库              日志数据               第三方数据
        ┌───────────┐         ┌──────────┐           ┌──────────┐
        │  MySQL    │         │  Nginx   │           │  微信支付  │
        │  Oracle   │──Sqoop──│  Kafka   │──Flume──  │  物流API  │
        │  MongoDB  │  Canal  │  Flink   │           │  天气数据  │
        └─────┬─────┘         └────┬─────┘           └────┬─────┘
              └─────────────────────┴─────────────────────┘
                                   │
                                   ▼
                         ┌─────────────────┐
                         │  Hive / Hudi    │
                         │  ODS 数据库      │
                         └─────────────────┘
```

### ⚠️ 设计注意事项

1. **分区策略**：通常按 `dt`（日期）分区，便于数据管理和历史追溯
2. **全量 vs 增量**：
   - 数据量小、变化频繁 → 全量同步（如用户基础信息）
   - 数据量大、新增为主 → 增量同步（如订单、日志）
3. **不做 JOIN**：ODS 层严禁关联操作，保持原子性
4. **压缩存储**：使用 ORC/Parquet + Snappy 压缩，节省存储空间

## 4. DWD — 数据明细层 (Data Warehouse Detail)

### 📌 定义

DWD（Data Warehouse Detail，数据明细层）是数据仓库的**清洗加工层**。它以 ODS 层数据为输入，通过**数据清洗、标准化、维度关联**，生成面向业务过程的、高质量的明细数据。

> 💡 **类比**：ODS 是"洗澡前的泥巴孩子"，DWD 是"洗干净后穿好衣服的孩子"——数据还是那个数据，但干净了、规范了。

### 🎯 核心作用

| 职责 | 具体操作 | 举例 |
|-----|---------|-----|
| **数据清洗** | 过滤异常值、空值处理 | 过滤金额为负数的订单 |
| **字段标准化** | 统一枚举值、日期格式 | `"male"/"M"/"男"` → `1`; `"20240101"` → `2024-01-01` |
| **维度退化** | 将维度字段嵌入事实表 | 将省份名称关联到订单行 |
| **脱敏处理** | 处理敏感信息 | 手机号中间四位替换为 `****` |
| **业务过滤** | 去除测试数据、内部账号 | 过滤员工账号的订单 |

### 🏗️ DWD 的数据模型设计

DWD 层遵循**维度建模**理论（Kimball 方法论），核心概念：

```
                    ┌─────────────────┐
                    │   事实表 (Fact)   │
                    │  dwd_trade_order │
                    │ ┌─────────────┐ │
                    │ │ order_id    │ │
                    │ │ user_id  ───┼─┼──→ dim_user (用户维表)
                    │ │ sku_id   ───┼─┼──→ dim_sku  (商品维表)
                    │ │ province_id─┼─┼──→ dim_province (地区维表)
                    │ │ order_amount│ │
                    │ │ create_time │ │     ← 度量值 (Measures)
                    │ └─────────────┘ │
                    └─────────────────┘

   事实表 = 业务过程的度量记录 (发生了什么)
   维表   = 描述业务实体的属性 (涉及谁/哪里/什么)
```

### 📂 典型表分类

**① 事务型事实表**（记录每一次业务事件）
```sql
-- 订单明细事实表
dwd_trade_order_detail_inc
  order_id        -- 订单ID
  order_detail_id -- 订单明细ID
  user_id         -- 用户ID（已关联维表）
  province_name   -- 省份名称（维度退化）
  sku_id          -- 商品ID
  sku_name        -- 商品名称（维度退化）
  order_price     -- 原始单价
  sku_num         -- 数量
  split_total_amount -- 分摊金额（已计算）
  dt              -- 分区日期
```

**② 周期快照事实表**（记录某一时刻的状态）
```sql
-- 用户每日登录行为快照
dwd_user_login_daily_snapshot
  user_id
  login_count
  last_login_time
  active_days_7d
  dt
```

### 🔧 DWD 层 ETL 示例（伪代码）

```sql
-- 从 ODS 清洗订单明细数据到 DWD
INSERT OVERWRITE TABLE dwd_trade_order_detail_inc
PARTITION (dt = '2024-01-15')
SELECT
    od.order_detail_id,
    od.order_id,
    od.user_id,
    od.sku_id,
    sk.sku_name,                    -- 维度退化：关联商品名称
    pr.province_name,               -- 维度退化：关联省份名称
    od.order_price,
    od.sku_num,
    od.order_price * od.sku_num AS split_total_amount,
    od.create_time
FROM ods_trade_order_detail_inc od
    JOIN ods_trade_order_full o ON od.order_id = o.order_id
    LEFT JOIN dim_sku_full sk ON od.sku_id = sk.sku_id
    LEFT JOIN dim_province_full pr ON o.province_id = pr.province_id
WHERE od.dt = '2024-01-15'
  AND od.order_price > 0               -- 清洗：过滤异常价格
  AND od.user_id NOT IN (              -- 清洗：过滤内部测试账号
      SELECT user_id FROM dim_internal_user
  );
```

## 5. DWM — 数据中间层 (Data Warehouse Middle)

### 📌 定义

DWM（Data Warehouse Middle，数据中间层）是数据仓库中的**轻度汇总层**，也叫**公共汇总层的前置**。它位于 DWD 和 DWS 之间，专门用于**跨业务域整合**和**中间指标的计算与沉淀**。

> ⚠️ **注意**：DWM 是一个**可选层**。小型数仓可以没有 DWM，由 DWD 直接进入 DWS。当业务复杂度上升，指标复用需求增加时，DWM 的价值才显现出来。

### 🎯 核心作用

```
        没有 DWM 时（指标孤岛问题）
        ┌──────┐  ┌──────┐  ┌──────┐
        │需求A  │  │需求B  │  │需求C  │
        │重复写 │  │重复写 │  │重复写 │
        │活跃用 │  │活跃用 │  │活跃用 │
        │户计算 │  │户计算 │  │户计算 │
        └──────┘  └──────┘  └──────┘
             ↑ 同一个指标被计算了3遍！逻辑不一致！

        有 DWM 后（指标统一）
        ┌────────────────────────────┐
        │  DWM：活跃用户指标 (计算一次) │
        └───────────┬────────────────┘
                    ├──→ 需求A 直接使用
                    ├──→ 需求B 直接使用
                    └──→ 需求C 直接使用
             ↑ 口径统一，避免"同一指标不同部门数字不一样"
```

### 🏗️ DWM 典型建设内容

**① 用户行为宽表**
```
dwm_user_action_wide
把用户在同一天的多种行为（浏览、搜索、下单、支付、退款）
横向拼接为一行，方便后续漏斗分析

user_id | page_view_count | search_count | order_count | pay_count | refund_count | dt
--------|----------------|--------------|-------------|-----------|--------------|----------
U001    |      45         |      8       |      3      |     2     |      0       | 2024-01-15
U002    |      12         |      2       |      0      |     0     |      0       | 2024-01-15
```

**② 跨域关联整合**
```
dwm_trade_pay_detail
= 交易域(订单) + 支付域(支付流水) + 促销域(优惠券)
整合后：每笔订单同时包含支付信息和优惠信息

好处：DWS 层不需要再做跨域 JOIN，性能提升明显
```

### 💡 DWM vs DWD：关键区别

```
┌─────────┬──────────────────────────┬──────────────────────────┐
│  维度    │         DWD              │          DWM             │
├─────────┼──────────────────────────┼──────────────────────────┤
│ 粒度     │ 最细粒度（每一行业务事件） │ 轻度汇总（按日/用户/商品聚合）│
│ JOIN     │ 事实表 + 维表             │ 跨事实表 JOIN（跨业务域）   │
│ 指标     │ 原始度量值               │ 衍生指标（转化率、留存率等） │
│ 复用性   │ 高（原始粒度最通用）       │ 高（公共中间指标）          │
└─────────┴──────────────────────────┴──────────────────────────┘
```

## 6. DWS — 数据服务层 (Data Warehouse Service)

### 📌 定义

DWS（Data Warehouse Service，数据服务层）是数据仓库的**主题汇总层**。它以业务主题（用户、商品、交易、流量等）为维度，对 DWD/DWM 层数据进行**高度汇总**，生成面向分析场景的**宽表和汇总指标**。

> 💡 **类比**：DWD 是超市货架上的每一件单品，DWS 是把货架上的商品按类别整理成的"购物手册"——按主题聚合，一目了然。

### 🎯 核心作用

1. **按主题域组织数据**：用户域、商品域、交易域、流量域等
2. **提供高性能查询基础**：数据已提前聚合，报表查询秒级响应
3. **指标统一口径**：DAU、GMV、转化率等核心指标在此计算完成
4. **支撑多种分析场景**：趋势分析、对比分析、下钻分析

### 📊 DWS 主题域与典型宽表

```
DWS 层
├── 用户主题域
│   ├── dws_user_action_daycount        -- 用户每日行为汇总
│   ├── dws_user_user_login_td          -- 用户登录累计指标
│   └── dws_user_order_daycount         -- 用户每日订单汇总
│
├── 商品主题域
│   ├── dws_trade_sku_order_daycount    -- 商品每日销售汇总
│   ├── dws_trade_sku_favor_daycount    -- 商品每日收藏汇总
│   └── dws_trade_sku_cart_daycount     -- 商品每日加购汇总
│
├── 交易主题域
│   ├── dws_trade_province_order_daycount  -- 地区每日交易汇总
│   ├── dws_trade_user_order_daycount      -- 用户每日交易汇总
│   └── dws_trade_activity_order_daycount  -- 活动效果汇总
│
└── 流量主题域
    ├── dws_traffic_page_visitor_daycount  -- 页面每日访问汇总
    └── dws_traffic_source_daycount        -- 流量来源汇总
```

### 🔧 DWS 宽表设计示例

```sql
-- 用户每日行为汇总宽表
-- dws_user_action_daycount

CREATE TABLE dws_user_action_daycount (
    user_id          BIGINT    COMMENT '用户ID',
    -- 登录行为
    login_count      BIGINT    COMMENT '登录次数',
    -- 浏览行为
    page_view_count  BIGINT    COMMENT '页面浏览次数',
    product_view_count BIGINT  COMMENT '商品详情浏览次数',
    -- 搜索行为
    search_count     BIGINT    COMMENT '搜索次数',
    -- 交易行为
    cart_add_count   BIGINT    COMMENT '加购次数',
    order_count      BIGINT    COMMENT '下单次数',
    order_amount     DECIMAL   COMMENT '下单金额',
    payment_count    BIGINT    COMMENT '支付次数',
    payment_amount   DECIMAL   COMMENT '支付金额',
    -- 售后行为
    refund_count     BIGINT    COMMENT '退款次数',
    refund_amount    DECIMAL   COMMENT '退款金额'
) COMMENT '用户每日行为汇总'
PARTITIONED BY (dt STRING);
```

### 📈 核心指标在 DWS 层的计算

```
GMV（商品交易总额）
= SUM(订单金额) WHERE 支付状态 = '已支付'
= 在 dws_trade_province_order_daycount 中预先计算好

DAU（日活跃用户数）
= COUNT(DISTINCT user_id) WHERE 登录次数 > 0
= 在 dws_user_action_daycount 中预先计算好

转化率
= 下单用户数 / 浏览商品用户数
= 基于 dws_user_action_daycount 中的多个字段组合计算
```

## 7. ADS — 应用数据层 (Application Data Store)

### 📌 定义

ADS（Application Data Store，应用数据层）是数据仓库的**最终输出层**，也是离业务最近的一层。它直接面向各类业务应用（报表、大屏、API、推荐系统），提供**开箱即用**的数据产品。

> 💡 **类比**：如果 DWS 是超市仓库里整理好的货物，ADS 就是**已经摆到货架上的商品**——消费者（业务方）取用即可，不需要再加工。

### 🎯 核心作用

| 应用场景 | ADS 输出形式 | 具体示例 |
|---------|------------|---------|
| **管理报表** | 结果宽表/视图 | 日销售报表、月度经营报表 |
| **数据大屏** | 实时/离线指标 | 实时 GMV 大屏、订单地图 |
| **运营系统** | 用户标签/人群包 | 高价值用户、流失预警用户 |
| **推荐系统** | 特征向量 | 用户兴趣特征、商品热度特征 |
| **风控系统** | 风险指标 | 用户异常行为评分 |
| **对外 API** | 数据接口 | 商家销售数据 API |

### 📂 ADS 典型表设计

```
ADS 层
├── 报表类
│   ├── ads_gmv_daycount              -- 每日GMV汇总
│   ├── ads_new_user_daycount         -- 每日新增用户
│   ├── ads_user_retention_daycount   -- 用户次日/7日留存
│   └── ads_funnel_conversion_rate    -- 购买漏斗转化率
│
├── 人群标签类
│   ├── ads_user_label_high_value     -- 高价值用户标签
│   ├── ads_user_label_churn_risk     -- 流失风险用户
│   └── ads_user_label_new_buyer      -- 首购用户
│
└── 特征类
    ├── ads_user_feature_for_recommend -- 推荐系统用户特征
    └── ads_item_feature_for_recommend -- 推荐系统商品特征
```

### 🔧 ADS 计算示例：用户次日留存率

```sql
-- ads_user_retention_daycount
INSERT INTO TABLE ads_user_retention_daycount
SELECT
    '2024-01-15'                      AS dt,
    COUNT(DISTINCT t1.user_id)        AS new_user_count,
    COUNT(DISTINCT t2.user_id)        AS retained_count,
    ROUND(
        COUNT(DISTINCT t2.user_id) * 100.0
        / NULLIF(COUNT(DISTINCT t1.user_id), 0), 2
    )                                 AS retention_rate
FROM
    -- 昨日新增用户
    (SELECT user_id FROM dws_user_action_daycount
     WHERE dt = '2024-01-14' AND login_count > 0
       AND user_id IN (SELECT user_id FROM dim_user WHERE create_date = '2024-01-14')
    ) t1
    -- 今日仍然活跃的用户
    LEFT JOIN (
        SELECT user_id FROM dws_user_action_daycount
        WHERE dt = '2024-01-15' AND login_count > 0
    ) t2 ON t1.user_id = t2.user_id;
```

### 🔄 数据流转全貌

```
数据源
  │
  │  ① 全量/增量同步（Sqoop/Canal/Flink CDC）

ODS（贴源存储）
  │

  ▼

  │

  ▼

  │
```

```
│ ODS  │ 原始粒度   │ 最大     │  低      │ 数据问题排查、重新跑数       │
│ DWD  │ 业务事件粒度│  大      │  中      │ 自定义分析、数据探查         │
│ DWM  │ 轻度聚合   │  中      │  中      │ 跨域指标复用                │
│ DWS  │ 主题日聚合 │  中小    │  高      │ 日常报表、定期分析           │
│ ADS  │ 结果粒度   │  小      │  极高    │ 实时大屏、业务系统对接       │
└──────┴───────────┴──────────┴──────────┴────────────────────────────┘
```

### 🔗 层间依赖关系

```
                  ◄── 数据来源方向 ──►

ODS ──────────────────────────────────────────►  ADS
            DWD ◄── ODS                     |
            DWM ◄── DWD (+ 跨域 DWD)        |

                                            |
 注意：ADS 可以跨层取数 (如取 DWD 做明细导出)  ◄──┘
```

## 10. 设计原则与最佳实践

### 📐 核心设计原则

#### 原则一：高内聚，低耦合

```
✅ 好的设计：
  每层只依赖其上一层，依赖关系清晰

  ODS ← DWD ← DWM ← DWS ← ADS
  （每层只依赖上层，改动影响可控）

❌ 坏的设计：
  ADS 直接读 ODS（跳层）
  DWS 和 DWD 相互引用（循环依赖）
```

#### 原则二：一次加工，多次复用

```
❌ 反模式（每个需求各写一套）：
  需求A：SELECT ... FROM ods_order JOIN dim_user ...（清洗逻辑重复）
  需求B：SELECT ... FROM ods_order JOIN dim_user ...（清洗逻辑重复）
  需求C：SELECT ... FROM ods_order JOIN dim_user ...（清洗逻辑重复）

✅ 正确方式（在 DWD 做一次，上层复用）：
  DWD：dwd_trade_order_detail（做一次清洗）
  需求A：SELECT ... FROM dwd_trade_order_detail（直接复用）
  需求B：SELECT ... FROM dwd_trade_order_detail（直接复用）
  需求C：SELECT ... FROM dwd_trade_order_detail（直接复用）
```

#### 原则三：明确各层的数据生命周期

```
┌──────┬──────────────┬────────────────────────┐
│  层  │  保留时长     │    原因                 │
├──────┼──────────────┼────────────────────────┤
│ ODS  │ 3~7 天       │ 数据量大，仅用于重新跑数  │
│ DWD  │ 1~3 年       │ 明细数据，支持历史分析    │
│ DWM  │ 1~3 个月     │ 中间结果，时效性适中      │
│ DWS  │ 3~5 年       │ 汇总数据，支持长期趋势    │
│ ADS  │ 按需（永久）  │ 面向业务，保留业务周期    │
└──────┴──────────────┴────────────────────────┘
```

### 🔧 常见坑与解决方案

#### 坑1："烟囱式"开发

```
问题：每个业务需求独立开发，形成数据孤岛
       需求A → 自己的清洗逻辑 → 自己的汇总逻辑
       需求B → 自己的清洗逻辑 → 自己的汇总逻辑
       结果：同一个"活跃用户"不同业务线统计出来数字不同！

解决：建立公共数据模型（DWD/DWM）+ 指标字典
      统一口径定义，所有需求引用同一份
```

#### 坑2：ODS 层直接用于生产查询

```
问题：数据未经清洗，含测试数据/异常数据，查询结果不可信
      且 ODS 保留期短，历史数据会消失

解决：生产查询一律从 DWD 层往上取数
      ODS 仅用于数据核查和重新跑数
```

#### 坑3：忽略数据血缘管理

```
问题：字段改名了，不知道影响哪些下游
      生产报表出错，不知道数据从哪来的

解决：使用 Apache Atlas / DataHub 等元数据管理工具
      记录每张表、每个字段的血缘关系
      形成完整的数据地图
```

### 🚀 从理论到实践的三个建议

1. **先建 ODS 和 DWD，保证数据质量是根基**
   - 没有高质量的 DWD，上层的所有分析都是建立在流沙上的

2. **按需建设 DWM，不要过度设计**
   - 业务初期可以跳过 DWM，当指标复用需求出现时再引入

3. **ADS 跟着业务走，数据驱动决策**
   - ADS 是数据价值的最终体现，要与业务方紧密对齐
